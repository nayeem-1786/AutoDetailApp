## Executive Summary

Four human-readable identifiers exist in the codebase, generated by **TWO DIVERGENT MECHANISMS**:

| Identifier | Format (actual) | Mechanism | Race-safe? | Test coverage of generator? |
|---|---|---|---|---|
| `quotes.quote_number` | **Q-XXXX** (4-digit, app) | (γ) Application-level read-max + INSERT | No — no retry | No (mocked in all 2 tests) |
| `transactions.receipt_number` | SD-XXXXXX (6-digit) | (β) PostgreSQL BEFORE INSERT trigger | No — no retry | No (mocked in tests) |
| `purchase_orders.po_number` | PO-XXXXXX (6-digit) | (β) PostgreSQL BEFORE INSERT trigger | No — no retry | No (no tests found) |
| `orders.order_number` | WO-XXXXX (5-digit, starts at 10001) | (γ) Application-level, deferred to payment success | No — no retry | No (mocked in tests) |

**The patterns diverge along two axes:** (1) generator location (DB trigger β vs. application γ), and (2) assignment timing (at-INSERT vs. deferred-to-payment-success). **Quote and receipt — the two existing pieces of the AC-10 spine — use OPPOSITE mechanisms.** Neither pattern carries race retry; both rely on the UNIQUE constraint to fail-fast.

**Format drift surfaced (mid-severity):** `quote_number`'s DB trigger (`Q-XXXXXX`, 6-digit) is shadowed by the application generator (`Q-XXXX`, 4-digit) which always supplies the column on INSERT — the trigger never fires in practice and would produce inconsistent numbers if it ever did.

**Convergence verdict for AC-10:** PATTERNS DIVERGE. Phase 3 Theme A faces a fork — either pick one existing mechanism and apply to `appointment_number` (accepting the parallel-tier accumulation), or refactor all four onto a unified pattern (one source of truth for identifier generation). Per the operator's locked principle ("best-in-class, no patch work"), the from-scratch option landscape is documented in Target E.

## Target A — quote_number current pattern

### A.1 — Format

**Actual production format:** `Q-XXXX` (Q-prefix + 4-digit zero-padded counter)

- Source: `src/lib/utils/quote-number.ts:27` — `return \`Q-${String(nextNum).padStart(4, '0')}\`;`
- Starting value: **1** → first quote is `Q-0001` (source: `quote-number.ts:18` — `let nextNum = 1;`)
- Padding: 4-digit zero-pad
- Prefix: `Q-` (no year segment, no other characters)

**Documented format vs actual:** SMS palette (`src/lib/sms/palette.ts:` & `src/lib/sms/sms-contracts.source.ts:`) documents `quote_number` sample as `"Q-001234"` (6-digit) — drift between documentation sample and actual production format. Customer-facing SMS receives whatever the actual column value is (`Q-XXXX`), so the drift is documentation-only, not functional.

**Format drift inside generator system itself:** The DB trigger `generate_quote_number()` at `supabase/migrations/20260201000037_create_functions_triggers.sql:79-92` produces `Q-XXXXXX` (6-digit, `LPAD(next_num::TEXT, 6, '0')`). Two different formats coexist in the system — see A.2 for why the trigger is dormant.

### A.2 — Generation mechanism

**Category: (γ) Application-level counter + (β) shadowed DB trigger**

Primary path — application-level, mechanism (γ):

```typescript
// src/lib/utils/quote-number.ts:6-28
export async function generateQuoteNumber(supabase?): Promise<string> {
  const client = supabase ?? createAdminClient();
  const { data } = await client
    .from('quotes')
    .select('quote_number')
    .order('quote_number', { ascending: false })
    .limit(1)
    .single();
  let nextNum = 1;
  if (data?.quote_number) {
    const match = data.quote_number.match(/^Q-(\d+)$/);
    if (match) {
      nextNum = parseInt(match[1], 10) + 1;
    }
  }
  return `Q-${String(nextNum).padStart(4, '0')}`;
}
```

Two callers — both supply `quote_number` in the INSERT payload (so the DB trigger's `WHEN (NEW.quote_number IS NULL)` clause evaluates FALSE → trigger never fires):

1. **Quote service (POS / admin / SMS-AI):** `src/lib/quotes/quote-service.ts:131` (call) → `:164` (insert payload includes `quote_number: quoteNumber`)
2. **Voice agent quotes endpoint:** `src/app/api/voice-agent/quotes/route.ts:213` (call) → `:248` (insert payload includes `quote_number: quoteNumber`)

Shadowed path — DB trigger, mechanism (β):

```sql
-- supabase/migrations/20260201000037_create_functions_triggers.sql:79-98
CREATE OR REPLACE FUNCTION generate_quote_number()
RETURNS TRIGGER AS $$
DECLARE next_num INTEGER;
BEGIN
  SELECT COALESCE(MAX(CAST(SUBSTRING(quote_number FROM 3) AS INTEGER)), 0) + 1
  INTO next_num
  FROM quotes
  WHERE quote_number LIKE 'Q-%';
  NEW.quote_number = 'Q-' || LPAD(next_num::TEXT, 6, '0');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tr_quote_number
  BEFORE INSERT ON quotes
  FOR EACH ROW
  WHEN (NEW.quote_number IS NULL)
  EXECUTE FUNCTION generate_quote_number();
```

The trigger is structurally dormant — every callsite supplies `quote_number` from the application layer, so the `WHEN` clause never permits it to fire. If it ever DID fire (e.g., a raw SQL INSERT bypassing the app), it would produce a 6-digit number conflicting with the 4-digit production format.

### A.3 — Race condition handling

**None.** Two concurrent `generateQuoteNumber` calls can:
1. Both read max (e.g., `Q-0099`)
2. Both compute next as `Q-0100`
3. Both attempt INSERT with `quote_number = 'Q-0100'`
4. First INSERT succeeds; second fails with PostgreSQL `23505` UNIQUE violation
5. Failing caller surfaces "Failed to create quote" — `src/lib/quotes/quote-service.ts:200-202` (`throw new Error('Failed to create quote')`) or `voice-agent/quotes/route.ts:266-269` (`return NextResponse.json({ error: 'Failed to create quote' }, { status: 500 })`)

No retry logic. No backoff. No reservation pattern. Grep confirms: zero references to PostgreSQL error code `23505`, `duplicate key`, `unique_violation`, `retry`, or related strings in any of the three generator callsites (quote-number.ts, quote-service.ts, voice-agent/quotes/route.ts).

The UNIQUE constraint on `quotes.quote_number` (`DB_SCHEMA.md:2082` — `quote_number TEXT UNIQUE NOT NULL`, indexed at `:2127`) is the only race safety, and it fails closed (errors the request).

### A.4 — Transactional behavior

**Hybrid: no gaps on simple rollback, BUT potential number reuse on items-error rollback path.**

Two distinct paths inside `createQuote` (`quote-service.ts`):

**Path 1 — quote INSERT fails directly** (`quote-service.ts:199-202`): the row is never inserted; `generateQuoteNumber` had returned a number but no row was written. The next call's `MAX()` reads the same prior max. **No gap, no reuse.**

**Path 2 — quote INSERT succeeds, items INSERT fails, cleanup DELETEs the quote** (`quote-service.ts:218-228`):
```typescript
if (itemsError) {
  await supabase.from('quotes').delete().eq('id', quote.id);
  throw new Error('Failed to create quote items');
}
```

After the DELETE, the row with `Q-0100` is gone. The next `generateQuoteNumber` call reads `MAX()` from `quotes` table — the just-deleted row is excluded — so the next call computes `Q-0100` again. **No gap, but the number Q-0100 is REUSED.** If downstream consumers observed the briefly-existing Q-0100 (webhooks at `quote-service.ts:233` `fireWebhook('quote_created', ...)` fire AFTER items-insert success, so the items-error path does NOT fire the webhook — partial mitigation), the reuse risk is bounded. Audit logs or other side-effects between the quote INSERT (`:194`) and items INSERT (`:218`) — if any — could still leak the reused number.

**Observation:** the items-INSERT cleanup runs AFTER the quote INSERT but BEFORE the webhook fire. The webhook only fires on success path (`:233`). So the reuse risk surfaces only if a non-webhook observer reads `quote_number` between INSERT-success-and-items-INSERT-failure — a tight window but non-zero.

### A.5 — Test coverage

**No race / gap / concurrency tests exist.** All test usage mocks the generator:

- `src/app/api/voice-agent/quotes/__tests__/route.test.ts` — `generateQuoteNumber: async () => 'Q-1001'` (constant-stub mock)
- `src/lib/quotes/__tests__/quote-service.modifiers.test.ts` — `generateQuoteNumber: vi.fn(async () => 'Q-TEST-001')` (constant-stub mock)

The actual generator at `src/lib/utils/quote-number.ts` has zero unit tests. No integration test exercises:
- Concurrent generation
- Race-then-UNIQUE-violation behavior
- Cleanup-rollback gap-vs-reuse behavior
- Empty-table starting-value semantics

## Target B — receipt_number current pattern

### B.1 — Format

**Actual production format:** `SD-XXXXXX` (SD-prefix + 6-digit zero-padded counter)

- Source: `supabase/migrations/20260201000037_create_functions_triggers.sql:45` — `NEW.receipt_number = 'SD-' || LPAD(next_num::TEXT, 6, '0');`
- Starting value: **1** → first receipt is `SD-000001` (source: `:40` — `COALESCE(MAX(...), 0) + 1`)
- Padding: 6-digit zero-pad
- Prefix: `SD-` (Smart Details abbreviation; no year segment)

**Documentation drift:** SMS palette documents `receipt_number` sample as `"R-001234"` (`src/lib/sms/palette.ts` / `sms-contracts.source.ts`) — wrong prefix and digit count. Documentation-only, not functional.

### B.2 — Generation mechanism

**Category: (β) PostgreSQL BEFORE INSERT trigger, conditional on NULL**

```sql
-- supabase/migrations/20260201000037_create_functions_triggers.sql:35-54
CREATE OR REPLACE FUNCTION generate_receipt_number()
RETURNS TRIGGER AS $$
DECLARE next_num INTEGER;
BEGIN
  SELECT COALESCE(MAX(CAST(SUBSTRING(receipt_number FROM 4) AS INTEGER)), 0) + 1
  INTO next_num
  FROM transactions
  WHERE receipt_number LIKE 'SD-%';
  NEW.receipt_number = 'SD-' || LPAD(next_num::TEXT, 6, '0');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tr_transaction_receipt_number
  BEFORE INSERT ON transactions
  FOR EACH ROW
  WHEN (NEW.receipt_number IS NULL)
  EXECUTE FUNCTION generate_receipt_number();
```

All three transaction-creating callsites leave `receipt_number` out of the INSERT payload (column defaults to `NULL`, trigger fires):

1. **POS live transactions** (`src/app/api/pos/transactions/route.ts:176-200`) — INSERT payload omits `receipt_number`
2. **POS offline transaction sync** (`src/app/api/pos/sync-offline-transaction/route.ts:71-97`) — INSERT payload omits `receipt_number`
3. **Booking deposit** (`src/app/api/book/route.ts:650-664`) — INSERT payload omits `receipt_number`; explicit comment at `:647` confirms: *"receipt_number is auto-generated by DB trigger (SD-XXXXXX format)"*

No application-level generator analog exists for `receipt_number`. Unlike `quote_number`, there is exactly one mechanism in play.

**Notable:** the `transactions.receipt_number` column is defined as `TEXT UNIQUE` (NULLABLE, per `DB_SCHEMA.md:2938` — no NOT NULL constraint). The nullability is what permits the trigger's `WHEN (NEW.receipt_number IS NULL)` gating to work — and also means a transaction CAN exist with a NULL `receipt_number` if some code path bypasses the trigger (e.g., raw SQL or a future code path that supplies `receipt_number=null` explicitly).

### B.3 — Race condition handling

**None.** The trigger's `SELECT MAX(...)` runs inside the row-INSERT transaction at PostgreSQL's default isolation level (READ COMMITTED). Two concurrent INSERTs:

1. Both fire the trigger
2. Both `SELECT MAX(...)` and read the same value (each sees committed rows; neither sees the other's uncommitted INSERT)
3. Both compute the same `next_num`
4. Both attempt to assign `SD-000050` (example)
5. First commit succeeds; second commit fails on the UNIQUE constraint (`DB_SCHEMA.md:2981` — `transactions_receipt_number_key`)
6. PostgreSQL rolls back the failing INSERT; the calling application surfaces the error

There is no retry inside the trigger function (PL/pgSQL `EXCEPTION WHEN unique_violation THEN RETRY` block is absent — verified by reading the full migration). Application code does not retry either (verified by reading all three callsites — none catch `23505` or implement retry logic).

The race window is small (microseconds — the time between the trigger's MAX read and the row commit) but architecturally real. Under sustained concurrent load (e.g., end-of-day batch close-out), this is the most likely race path.

### B.4 — Transactional behavior

**No gaps on rollback.** The trigger fires inside the BEFORE INSERT of the row. If the row's INSERT is rolled back (either explicitly via the outer transaction or implicitly by a constraint failure), the trigger's assignment is rolled back with it. The next INSERT's `MAX()` query will not see the rolled-back row, so the same number can be re-used by a subsequent successful INSERT. **No gap, no observable reuse** — because the rolled-back row never made it into a committed snapshot any external observer could read.

This is structurally cleaner than the application-level γ pattern's rollback behavior (Target A.4). The trigger keeps the number+row binding inside one transactional unit; the application pattern in `createQuote` separates them across two statements with an intermediate DELETE on items-error.

### B.5 — Test coverage

**No race / gap / concurrency tests exist.** All references mock or hardcode the receipt number:

- `src/app/api/pos/transactions/__tests__/auto-receipt-interlock.test.ts:181` — `receipt_number: 'SD-001'` (constant-stub fixture)
- All receipt-baseline fixtures (`src/lib/data/__tests__/__fixtures__/receipt-baselines/*`) supply pre-set receipt numbers, not trigger-generated ones

The DB trigger itself has zero direct tests. No integration test exercises:
- Concurrent transaction creation under load
- Trigger behavior on transaction rollback
- Trigger behavior when `receipt_number` is explicitly NULL vs. omitted from payload
- What happens if a code path supplies `receipt_number = 'SD-...something invalid...'` (the trigger's `WHEN` clause would skip the auto-generation, accepting whatever was supplied)

## Target C — other identifier patterns

Two additional human-readable identifier systems exist in the codebase. Both relevant as comparative context for AC-10's `appointment_number` design.

### C.1 — `purchase_orders.po_number`

**Format:** `PO-XXXXXX` (6-digit, same shape as receipt_number)

**Mechanism: (β) PostgreSQL BEFORE INSERT trigger** — `supabase/migrations/20260201000037_create_functions_triggers.sql:57-76`. Same exact shape as `generate_receipt_number()` — read-max + LPAD + supply if NULL.

**Confirmed callsite behavior:** `src/app/api/admin/purchase-orders/route.ts:125` — explicit comment *"Create purchase order (po_number auto-generated by trigger)"*; the INSERT payload omits `po_number`. Same one-path pattern as receipt_number.

**Race / gap / test coverage:** identical to receipt_number (no retry, no gap on rollback, no tests of the generator).

**Customer-facing?** No — internal-only (admin purchasing UI). Not surfaced to customers, not in SMS templates, not in receipts.

### C.2 — `orders.order_number`

**Format:** `WO-XXXXX` (5-digit, starts at **10001**) — different from all other identifiers in this system.

**Mechanism: (γ) Application-level read-max + INSERT, ASSIGNMENT DEFERRED TO PAYMENT SUCCESS.** This is the most architecturally distinct pattern in the four-identifier set.

```typescript
// src/lib/utils/order-number.ts:9-32
export async function generateOrderNumber(supabase?): Promise<string> {
  const client = supabase ?? createAdminClient();
  const { data } = await client
    .from('orders')
    .select('order_number')
    .not('order_number', 'is', null)
    .order('order_number', { ascending: false })
    .limit(1)
    .single();
  let nextNum = 10001;
  if (data?.order_number) {
    const match = data.order_number.match(/^WO-(\d+)$/);
    if (match) nextNum = parseInt(match[1], 10) + 1;
  }
  return `WO-${String(nextNum).padStart(5, '0')}`;
}
```

**Deferred-assignment design:** orders are created at the start of checkout (without an `order_number`), and the number is assigned only when the Stripe webhook confirms payment success (`src/app/api/webhooks/stripe/route.ts:250-261`). Until paid, orders exist as orphans with `order_number IS NULL` — the generator's `.not('order_number', 'is', null)` filter at line 17 explicitly excludes these orphans from the MAX read.

**Rationale (inferred from comments):** *"Orders by order_number DESC to prevent reuse. Filters out NULL order_numbers (pending orders without assigned numbers)."* — keeps the customer-facing identifier reserved for orders that actually completed payment, avoiding "we cancelled your order WO-12345" friction for never-paid checkout sessions.

**Race / gap / test coverage:** same γ-pattern weaknesses as `quote_number` — no retry, race possible at high concurrency. No tests of the generator directly.

**Starting value:** 10001 (not 1) — chosen to keep visible identifiers consistently 5-digit and to avoid the visual "first customer ever" connotation of WO-00001.

### C.3 — Identifiers NOT found

- **`refunds`** — no human-readable identifier (UUID `id` only; `stripe_refund_id` is a Stripe identifier, not Smart Details' own). Not in scope for AC-10 but worth noting that refund display surfaces (POS Transactions tab, customer email) reference the parent transaction's `receipt_number` rather than introducing a refund-specific identifier.
- **`appointments`** — no human-readable identifier today. This is the GAP AC-10 closes.
- **`jobs`** — no human-readable identifier today. Per AC-10's spine principle, jobs inherit the appointment's number.
- **`customers`** — no human-readable identifier (no customer number system; identification is by phone/email).
- **`invoices`** — no separate invoice entity; transactions serve as both.

### Cross-pattern observations

1. **Mechanism split is 2:2 — not converged.** Quote + Order use γ (application); Receipt + PO use β (DB trigger). The split does NOT cleanly track any obvious axis (customer-facing vs internal: Quote is customer-facing γ, Receipt is customer-facing β; Order is customer-facing γ, PO is internal β — no pattern).
2. **Inferred historical sequence:** the migration that introduced the DB triggers (`20260201000037`) created triggers for receipt, PO, AND quote — suggesting an original architectural intent of "all identifiers via DB trigger." The quote app-level generator at `src/lib/utils/quote-number.ts` appears to have been introduced LATER and bypassed the trigger by always supplying the column. The dormant trigger is the evidence of architectural drift.
3. **Assignment timing is unique to `order_number`.** Three of four identifiers assign at row-creation; one defers to payment success. This deferred-assignment pattern is a deliberate UX choice (only show identifiers to customers when the transaction is real), but it adds an axis of complexity not present in the other three.
4. **NULLABLE columns vs NOT NULL is inconsistent.** Quote + PO are NOT NULL; Receipt + Order are NULLABLE. The NULLABLE columns require the DB trigger's `WHEN (NEW.X IS NULL)` gate pattern; the NOT NULL columns COULD use a trigger but quote bypasses it via the app generator.
5. **None of the four generators carry concurrency tests.** AC-10's `appointment_number` will inherit the test gap unless Phase 3 Theme A explicitly addresses it.

## Target E — best-in-class assessment + from-scratch option landscape

Per the operator's locked principle (*"best-in-class, no patch work. Invest the time NOW to get it right"*), this section weighs both directions honestly without recommendation.

### E.1 — Is there an existing "best" pattern across the four?

**No clear winner.** Both γ and β have meaningful weaknesses:

**(γ) Application-level — Quote & Order:**
- **Pros:** flexible (can defer like Order does; can format-customize per call); generator behavior is visible/testable in TS without DB introspection; no DB function to maintain.
- **Cons:** race-fragile (read-max → INSERT is two statements with a TOCTOU window); the items-cleanup path in Quote introduces NUMBER REUSE risk (Target A.4); requires every callsite to remember to call the generator (drift risk if a new callsite is added without it).

**(β) DB BEFORE INSERT trigger — Receipt & PO:**
- **Pros:** automatic (callers don't need to remember); rollback-clean (no gap, no reuse); one place to maintain the format logic; trigger fires for any INSERT path including raw SQL.
- **Cons:** still race-fragile at default READ COMMITTED isolation (concurrent INSERTs can compute same MAX); harder to test (requires DB-level setup); behavior is invisible in the application code path.

**Neither pattern is race-safe.** Both rely on the UNIQUE constraint to fail-fast under contention. For Smart Details' production load profile (handful of concurrent operators, occasional batch operations), the failure rate is likely negligible — but it's a load-bearing assumption that's never been measured.

**If forced to pick "existing best": (β) DB trigger is structurally cleaner** for the AC-10 use case because:
1. Rollback behavior is transactionally atomic (no number reuse window like Quote's items-cleanup path)
2. New code paths automatically work (operator-mediated convert, walk-in atomic create, booking flow, voice-agent quote-to-appointment — all would get `appointment_number` for free without remembering to call a generator)
3. Trigger pattern matches 2/4 existing identifiers (receipt + PO); applying to appointment would tilt the split to 3/4 trigger, 1/4 app
4. Already documented as the intended pattern by the dormant `generate_quote_number()` trigger in the same migration

**Caveat on "(β) is best for AC-10 specifically":** the format-drift bug between `quote_number`'s dormant trigger (Q-XXXXXX) and the actual production format (Q-XXXX) is a maintainability hazard that COPYING the β pattern would inherit. Phase 3 Theme A would need to actively delete the dormant `generate_quote_number()` trigger as part of any AC-10 implementation to avoid leaving a third trigger that's similarly latent.

### E.2 — What would a "from-scratch best-in-class" approach look like?

If designing the four identifiers fresh today, the option landscape:

#### Option Z1 — Per-entity DB trigger (current β pattern, extended)

Apply the existing β trigger pattern to `appointment_number`. Delete the dormant `generate_quote_number()` trigger. Leave the app-level `quote_number` and `order_number` generators in place.

- **Effort:** lowest. One new trigger + DROP TRIGGER for the dormant one.
- **Risk:** lowest — incremental, no data migration on existing identifiers.
- **Trade-off:** the architectural divergence (Quote + Order use γ, Receipt + PO + Appointment use β) is preserved. Future maintainers face the same "why are these different?" question that this audit just answered.

#### Option Z2 — Unified DB function `next_identifier(entity_type text) RETURNS text`

Centralize all four (eventually five) identifier-generation logic in a single PostgreSQL function:

```sql
-- Sketch (NOT a recommendation, just shape):
CREATE FUNCTION next_identifier(entity_type text) RETURNS text AS $$
DECLARE
  prefix text;
  pad_width int;
  start_at int;
  next_num int;
BEGIN
  CASE entity_type
    WHEN 'quote' THEN prefix := 'Q-'; pad_width := 4; start_at := 1;
    WHEN 'receipt' THEN prefix := 'SD-'; pad_width := 6; start_at := 1;
    WHEN 'po' THEN prefix := 'PO-'; pad_width := 6; start_at := 1;
    WHEN 'order' THEN prefix := 'WO-'; pad_width := 5; start_at := 10001;
    WHEN 'appointment' THEN prefix := 'A-'; pad_width := 5; start_at := 10001;  -- per AC-10
    ELSE RAISE EXCEPTION 'Unknown entity_type: %', entity_type;
  END CASE;
  -- ... read max from appropriate table, compute next, return formatted
END;
$$ LANGUAGE plpgsql;
```

Callers invoke from triggers OR application code (`SELECT next_identifier('appointment')` returns the next value).

- **Effort:** medium. One function + 4-5 callers (triggers and/or app generators) refactored. No data migration.
- **Risk:** medium — needs careful introduction; existing quote_number app generator would be deleted and quotes migrated to trigger-only.
- **Trade-off:** centralizes format logic (one place for AC-10 spine identifier format choices); does NOT solve the race problem (still uses `MAX()` read internally); makes test setup easier for the unified shape.

#### Option Z3 — Shared `identifier_sequences` table with type discriminator

A dedicated counter table that holds one row per entity type:

```sql
-- Sketch (NOT a recommendation):
CREATE TABLE identifier_sequences (
  entity_type text PRIMARY KEY,
  current_value bigint NOT NULL DEFAULT 0,
  prefix text NOT NULL,
  pad_width int NOT NULL,
  updated_at timestamptz NOT NULL DEFAULT now()
);

-- Race-safe via row-level lock:
CREATE FUNCTION next_identifier(entity_type text) RETURNS text AS $$
DECLARE
  seq identifier_sequences;
BEGIN
  UPDATE identifier_sequences
  SET current_value = current_value + 1, updated_at = now()
  WHERE identifier_sequences.entity_type = next_identifier.entity_type
  RETURNING * INTO seq;
  RETURN seq.prefix || LPAD(seq.current_value::text, seq.pad_width, '0');
END;
$$ LANGUAGE plpgsql;
```

- **Pros:** RACE-SAFE (UPDATE ... RETURNING acquires a row-level lock, sequencing concurrent calls). Gaps still possible on rollback (UPDATE commits the counter increment even if the surrounding INSERT rolls back). Single source of truth for all identifier metadata (prefix, padding, current value).
- **Cons:** introduces a new hot row per entity type (the `identifier_sequences` row for `appointment` will be touched on every appointment creation — small but real contention point). Adding a new identifier type requires INSERTing a row, not just defining a column.
- **Effort:** highest. Requires data backfill (compute current max for each existing identifier, seed `current_value`). Risk during migration: any concurrent INSERT during the backfill window could race the seed.
- **Trade-off:** PROPER race safety; gaps allowed (acceptable per the trade-off operator might lock).

#### Option Z4 — PostgreSQL SEQUENCE (mechanism α)

Use a true PG `SEQUENCE` for each entity type:

```sql
-- Sketch (NOT a recommendation):
CREATE SEQUENCE appointment_number_seq START WITH 10001;
ALTER TABLE appointments
  ADD COLUMN appointment_number text GENERATED ALWAYS AS
    ('A-' || LPAD(nextval('appointment_number_seq')::text, 5, '0'))
  STORED UNIQUE;
```

- **Pros:** PG sequences are RACE-SAFE by design (atomic `nextval`); zero application code. Numbers are reserved as soon as `nextval` runs (gaps on rollback are inherent to sequences — accepted PG behavior).
- **Cons:** gaps on rollback are GUARANTEED (sequences don't roll back). For Smart Details' "best-in-class" operator framing, this might be considered a trade-off worth accepting (gaps are visible to operators but not to customers); for a "no gaps ever" preference, this option is excluded.
- **Effort:** medium. New sequences + column definitions. No retroactive backfill (each table's existing identifiers stay as-is; new rows use the sequence).
- **Trade-off:** simplest race-safe option, BUT introduces gaps as a permanent feature.

### E.3 — Migration implications

**If Option Z1 is chosen** (incremental — add trigger for appointment, delete dormant quote trigger):
- One new migration: `CREATE FUNCTION generate_appointment_number()` + `CREATE TRIGGER`.
- One same-migration cleanup: `DROP TRIGGER tr_quote_number; DROP FUNCTION generate_quote_number();`.
- No backfill required (appointments table currently has NO `appointment_number` column; introducing it with default trigger will only affect new rows).
- Risk: minimal.

**If Option Z2 is chosen** (unified function):
- One new migration introducing `next_identifier()`.
- Existing app generators (`quote-number.ts`, `order-number.ts`) refactored or deleted.
- Existing β triggers refactored to call `next_identifier()`.
- Data migration: NONE (the existing identifiers stay; the unified function continues their sequences).
- Risk: medium (touches all four identifier systems simultaneously).

**If Option Z3 is chosen** (shared sequences table):
- Migration introduces `identifier_sequences` table with one row per entity type.
- Backfill: `INSERT INTO identifier_sequences (entity_type, current_value, ...) VALUES ('quote', (SELECT MAX(CAST(SUBSTRING(quote_number FROM 3) AS INTEGER)) FROM quotes), ...), ...` — for all four existing identifiers.
- BACKFILL RACE: any concurrent identifier-generating call during the backfill window would race the seed. Mitigations: run during low-traffic window OR lock the relevant tables during seed.
- Existing app generators replaced; existing β triggers replaced.
- Risk: high — touches everything, requires data backfill, has a tight backfill race window.

**If Option Z4 is chosen** (PG sequences):
- One new migration per entity type: `CREATE SEQUENCE` + `ALTER TABLE ... ADD COLUMN` for appointment.
- For existing identifiers, two choices:
  - (a) Leave existing identifiers on current mechanism — only appointment uses sequence. (Divergence preserved.)
  - (b) Migrate existing to sequences: `CREATE SEQUENCE quote_number_seq START WITH (SELECT MAX(...) + 1 FROM quotes)`. No row backfill; only forward-looking.
- Risk: low for (a), medium for (b).

### F.2 — `appointment_number` format — confirm A-XXXXX, 5-digit, starts at 10001?

- **Starting value:** WO starts at 10001 to avoid "first-ever" connotation; SD/Q/PO start at 1. Which precedent should A follow?

**No pre-resolution.** This is operator preference + minor UX implication.

### F.3 — Should gaps in numbering be tolerated or eliminated?

Each pattern has different gap semantics:
- γ (Quote, Order) — no gap on simple failure, but **number REUSE risk** on Quote's items-error cleanup
- β (Receipt, PO) — no gap, no reuse (cleanest)
- Z3 shared-counter — gaps on rollback (UPDATE counter is independent of row INSERT)
- Z4 PG sequence — **gaps guaranteed** by sequence semantics

Operator preference question — "is a gap-free identifier sequence (Q-0099, Q-0100, Q-0101, ...) a hard requirement, or are gaps (Q-0099, Q-0103, ...) acceptable?" — directly constrains the option set.

**No pre-resolution.** Operational/cosmetic preference.

### F.4 — Should race-safety be designed in now, or accepted as a known gap?

Current load profile makes the race window unlikely to surface in production. AC-10 could lock identifier generation without addressing race safety (existing patterns all share the gap; appointment would be no worse).

OR — Phase 3 Theme A could close the race-safety gap as part of the unification work (only Z3 closes it without gap trade-offs; Z4 closes it but introduces gaps).

**No pre-resolution.** Operator decides whether race-safety is "best-in-class" worthy of the migration cost.

### F.5 — Should `appointment_number` also serve as the job identifier (no `job_number` column), or should jobs get their own?

- A derived job suffix (`A-12345-J` etc.) for operator clarity
- Just the job UUID's first 8 chars (current admin pattern at `src/app/admin/jobs/page.tsx` likely)

**No pre-resolution.** Phase 3 Theme A scope question.

### F.6 — Should the dormant `generate_quote_number()` trigger be deleted as part of AC-10 work?

The trigger's format (`Q-XXXXXX`, 6-digit) conflicts with the actual production format (`Q-XXXX`, 4-digit). If it ever fires (raw SQL path), it produces inconsistent numbers.
